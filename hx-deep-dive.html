Phases

工作流阶段

CLI 以阶段组织能力，而非单个命令。

Phase 01-03

准备输入

hx doc、hx plan、hx ctx 负责准备输入。

Phase 04-06

执行与校验

hx run、hx gate、hx review、hx fix 负责执行与复核。

Phase 07-08

沉淀与交付

hx entropy 与 hx mr 负责沉淀与交付。

Runtime

运行时路径

CLI 将资源划分为框架层、用户层和项目层。

层级 典型路径 角色
FRAMEWORK_ROOT 包内 src/ / docs/ / profiles/ 内置模板、内置 Profile、框架脚本。是 CLI 自身的只读资源来源。
USER_HX_DIR ~/.hx/ 跨项目共享的用户级配置和 Profile。由 hx setup 初始化。
PROJECT_ROOT <project>/.hx/、AGENTS.md 业务项目自己的配置、任务文档、计划文件、命令模板和 skills 入口。
i
配置优先级：内置默认值 → ~/.hx/config.json → .hx/config.json。
Profiles

Profile 解析

Profile 决定架构层级、门控命令、黄金规则与模板资源。

搜索顺序

Profile 搜索根目录

<project>/.hx/profiles/
~/.hx/profiles/
框架内置 profiles/

同名 Profile 按该顺序覆盖。

推断顺序

hx run 如何确定 profile

--profile 参数
→ progress.json 中的 profile
→ task-id 前缀推断
→ .hx/config.json.defaultProfile

计划文件中的 profile 与显式参数不一致时，hx run 会拒绝执行。

!
参数范围：只有读取 Profile 的命令需要 --profile；hx version、hx setup、hx done 不依赖该参数。
Context

上下文装配

hx run 通过固定清单组装上下文与 Hook。

主上下文

hx run 默认读取的文件

AGENTS.md
docs/golden-principles.md
当前 Profile 的 golden-rules.md
当前 Profile 的 profile.yaml
docs/plans/<feature>.md
docs/requirement/<feature>.md

路径由计划文件、Profile 解析结果与上下文解析器决定。

Hook 注入

两种定义方式，统一合并

约定式:
~/.hx/hooks/run-pre.md
.hx/hooks/run-pre.md

配置式:
{
  "hooks": {
    "run": { "pre": [".hx/hooks/extra.md"] }
  }
}

加载顺序为全局约定 → 项目约定 → 配置式数组；相同绝对路径会去重。

Quality

质量控制

质量控制由多个独立命令组成，可按场景组合执行。

静态校验

hx ctx / hx check

hx ctx 检查文档和 Profile 资源；hx check 串行执行 ctx + gate。

执行门控

hx gate

按固定顺序执行 lint → build → type → test → arch，只跑 Profile 里实际定义过的步骤，遇到失败立即停止。

复核修复

hx review / hx fix

review 输出检查清单；fix 根据日志或测试输出生成修复 Prompt。

!
占位符参数：若门控命令包含 {scheme} 等占位符，必须通过 CLI 参数显式传入。
Progress

计划与进度

多个命令依赖 docs/plans/<feature>-progress.json 进行状态推断。

计划生成

hx plan 输出两份文件

输出包括计划 Markdown 和 progress JSON。

状态流转

任务状态至少涉及这些值

pending
in-progress
done

hx done 更新任务状态；hx ctx 检查任务定义完整性。
