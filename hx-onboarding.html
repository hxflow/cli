First Look

命名与职责

区分产品名、命令名和包名。

概念层

产品名与命令名

HXFlow 是产品名，终端命令为 hx，npm 包名为 @harness-workflow/cli。

使用层

hx 的职责

负责组织文档、计划、Profile 规则与上下文，并生成执行 Prompt 或运行校验命令。

i
顺序：执行链路 → 运行机制 → 配置参考。
Install

安装

安装 npm 包后执行 hx setup 初始化用户级资源。

终端操作

安装与验证

# 全局安装 CLI
npm install -g @harness-workflow/cli

# 查看版本
hx version

# 部署全局资源
hx setup
资源写入

hx setup 写入目录

~/.hx/config.json
~/.hx/profiles/
~/.claude/commands/
~/.claude/skills/

这些目录承载跨项目共享的用户级资源。

Init

项目初始化

在项目根目录执行 hx init；跨目录场景使用 --target。

# 当前目录初始化
hx init --profile frontend

# 指定目标目录初始化
hx init --profile backend --target ../my-service
输出

标准目录与入口文件

.hx/config.json
docs/requirement/
docs/plans/
AGENTS.md
.CLAUDE.md
.claude/commands/
.claude/skills/
约束

AGENTS.md 默认在项目根目录

CLI 默认并只使用项目根目录的 AGENTS.md。

Workflow

执行链路

执行顺序如下。

01
创建需求文档
执行 hx doc user-login --profile frontend 创建需求文档；可用 --task <id> 写入任务号。
02
生成执行计划
补全验收标准后执行 hx plan user-login --profile frontend，生成计划 Markdown 和 *-progress.json。
03
上下文校验
执行 hx ctx --profile frontend，校验 AGENTS.md、需求文档、计划文件和 Profile 资源。
04
生成执行 Prompt
执行 hx run user-login TASK-FE-01 --profile frontend 生成 Prompt。
05
门控通过后标记完成
执行 hx gate --profile frontend 运行门控，通过后用 hx done TASK-FE-01 回写进度。
hx doc user-login --profile frontend
hx plan user-login --profile frontend
hx ctx --profile frontend
hx run user-login TASK-FE-01 --profile frontend
hx gate --profile frontend
hx done TASK-FE-01
✓
基线检查：执行前运行 hx ctx，校验文档、Profile 与 AGENTS.md 一致性。
Pitfalls

常见问题

以下错误会直接影响执行结果或上下文一致性。

错误 01

执行了 hx plan，却没先补需求文档

计划直接依赖当前需求文档内容。不完整的 AC 会被固化到计划和进度文件。

错误 02

误将 hx run 视为代码执行器

hx run 生成 Prompt，不直接修改代码。

错误 03

没有先执行 hx setup

缺少用户级 Profile、命令模板和 skills，会导致能力不完整。

错误 04

认为所有命令都要手动传 --profile

部分命令可自动推断 profile；初始化阶段应显式传入。

错误 05

任务做完了忘记 hx done

这会导致 *-progress.json 与实际状态不一致，并影响后续执行与交付。
